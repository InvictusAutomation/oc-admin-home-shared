# Skill Builder - AI Skill 创建与优化

基于 CodeBuddy 的 Skill 设计方法论，将人类工程经验编码为 AI 可执行的 SOP。

## 核心公式

```
Skill = SOP（步骤定义）
      + References（输出样本/领域知识）
      + Progress（状态持久化/断点恢复）
      + Auto-Continue（无人值守循环）
```

## 三层知识架构

| 层级 | 作用 | 示例 |
|------|------|------|
| **SOP** | 告诉 AI "怎么做" | 6步工作流、操作粒度定义 |
| **References** | 约束输出格式 | 示例样本、领域知识文档 |
| **Source Data** | 提供素材 | JSON、源码、API 响应 |

## SOP 设计原则

### 6 步标准工作流

```
┌─────────────┐    ┌─────────────┐    ┌─────────────┐
│ 1. 选取目标  │───>│ 2. 定向采集  │───>│ 3. 定向采集  │
│ (读进度文件) │    │ (数据源A)    │    │ (数据源B)    │
└─────────────┘    └─────────────┘    └─────────────┘
                                              │
                                              ▼
┌─────────────┐    ┌─────────────┐    ┌─────────────┐
│ 6. 自动继续  │<───│ 5. 更新进度  │<───│ 4. 生成输出  │
│ (处理下一个) │    │ (写进度文件) │    │ (参照样本)   │
└─────────────┘    └─────────────┘    └─────────────┘
```

### 关键设计决策

1. **小批量、高频次** - 批量越大，出错概率指数级增长
2. **外部脚手架** - 用进度文件维持跨次调用连续性
3. **自动继续机制** - 完成一个自动开始下一个，无需人工确认

## Progress 状态文件

```json
[
  { "name": "组件A", "status": "done" },
  { "name": "组件B", "status": "done" },
  { "name": "组件C", "status": "pending" }
]
```

**价值**：
- 断点恢复 - 中断后可继续
- 进度可视 - 知道做到哪了
- 并行协调 - 多批次处理同一任务

## References 参考样本

### 目录结构

```
skill/
├── SKILL.md              ← 第一层：SOP
├── references/
│   ├── example-output.json    ← 完整输出样本
│   └── domain-knowledge.md    ← 领域知识文档
└── scripts/              ← 第三层：可执行工具
    └── validator.js
```

### 样本设计原则

- 提供"标准答案的样子"
- 覆盖全部边缘情况
- 包含 unmapped 字段的标准写法

## 领域知识注入

### 四类知识缺口

| 类型 | 说明 | 注入方式 |
|------|------|----------|
| 设计模式 | Figma/React 的概念差异 | references 文档 |
| 状态拆分 | state → disabled + CSS伪类 | SOP 预定义规则 |
| 值语义 | INSTANCE_SWAP 的含义 | domain-knowledge.md |
| 映射边界 | 什么不存在于对端 | SOP 通用规则 |

### 按需注入原则

> 领域知识不应该"一股脑"全部灌给 AI，而应该结构化组织，在 AI 执行到相应步骤时"按需加载"。

## 渐进式处理飞轮

```
处理组件 N → 发现新模式 → 编码进 Skill → 自动应用到 N+1, N+2...
```

这种"在做中学、在学中编码、编码后复用"的循环，是"暴力投喂"无法实现的。

## 应用场景

- 批量文档生成
- 代码迁移
- 结构化数据处理
- 设计↔代码双向映射
- 大规模 API 集成

## 核心教训

**AI 不是"更快的打字员"**。

让 AI 一次性处理大量任务，本质上是把它当作"能读更多数据的人"来用。但 AI 的推理质量与输入规模之间存在**非线性衰减**：

```
质量 ▲
     │●●
     │   ●●
     │      ●●
     │         ●●●●●
     └──────────────────▶ 任务数量
```

正确做法：
1. 先花 10 分钟想清楚 SOP
2. 准备 References 约束格式
3. 设计 Progress 断点机制
4. 小批量、高频次处理
